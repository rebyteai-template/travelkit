# TripDesk 嵌入对接文档（iframe）

面向：接入方研发。
目标：把 TripDesk（AI 机票预订工作台）作为一个 iframe 嵌进贵方管理后台，用贵方的员工身份直接使用，无需二次登录。

线上地址：`https://tripdesk.impo.ai`

---

## 1. 一句话原理

贵方后台在渲染 TripDesk 标签页时，把**当前员工的身份**通过 URL 片段传给我们：

```
https://tripdesk.impo.ai/#uid=<员工稳定ID>&token=<员工 travelkit token>&k=<embed key>
```

我们用 `uid` 作为该员工的租户身份（订单/对话历史按它隔离），用 `token` 作为该员工调用 travelkit 的凭证（在其专属沙箱内跑机票工具）。

---

## 2. 接入方式

```html
<iframe
  src="https://tripdesk.impo.ai/#uid=EMP10086&token=TK_xxxxx&k=THE_EMBED_KEY"
  style="width:100%; height:100%; border:0"
  title="TripDesk"
  allow="clipboard-write"
></iframe>
```

### 三个参数

| 参数 | 含义 | 是否稳定 | 由谁提供 |
|------|------|----------|----------|
| `uid` | 员工**稳定**唯一标识（如工号/内部 user id） | **稳定**，同一个人永不变 | 接入方 |
| `token` | 该员工的 **travelkit token**（用于跑机票接口） | **会变**：员工 logout 再登录后是新 token | 接入方 |
| `k` | TripDesk 的 **embed 门禁 key**（全局共享密钥） | 固定 | **TripDesk 方单独提供**（不在本文档明文） |

### ⚠️ 必须用 URL 片段（`#`），不要用 query（`?`）

`#` 后面的内容**不会进入服务端日志、不会出现在 Referer 头**，浏览器拿到后由前端 JS 读取、随即从地址栏抹掉。把 `token` / `k` 放 `?` 查询串会泄漏到各类日志，**禁止**。

### token 轮换（员工重新登录）

- `uid` 不变、`token` 变 → **重新渲染 iframe**（带同一个 `uid` + 新 `token`）即可。
- 不要因为换了 token 就换 `uid`——那样该员工会丢失自己的对话/订单上下文。

---

## 3. 运行行为（接入方需知道）

- **首次发消息**会为该 `uid` 开通一个专属沙箱（约 1 分钟），之后复用，不会每次都开。
- 身份**只存在浏览器会话内**（sessionStorage）：刷新（F5）保持；关标签页/新开标签会清空，靠贵方下次渲染 iframe 时重新带上 `#uid&token&k` 恢复。我们这边**没有服务端会话**，每个请求都凭 URL 带来的身份现认。
- **支付为沙箱/演示模式**：助手可发起支付并返回第三方支付链接给用户自行完成，**绝不替用户付款、也绝不谎称已付**。
- **写操作红线**：下单/支付/退改等每一步都要用户在界面上明确确认。
- 默认简体中文。

---

## 4. 未授权页面

打开时若缺少 `k`（或 `k` 错误、缺 `uid`），TripDesk 会：
- 接口侧返回 `401`；
- 前端展示「无法访问 TripDesk，请从企业后台内打开」的提示页（不会暴露任何功能）。

---

## 5. 安全说明（当前阶段 & 后续）

**当前（联调/试用阶段）**：
- `k` 是全局共享门禁 key，挡住「只知道域名的陌生人」——没有 `k` 无法启动任何会话/沙箱。
- 但 `uid` 目前是**裸传、未签名**：拿到任意一条完整链接的人，理论上可以把 `uid` 改成别的同事的值来访问其数据。**所以现阶段请把带 `token`/`k` 的链接当作敏感凭证，仅在贵方后台服务端渲染、不要外泄。**

**正式上线前建议升级为签名握手**（消除上面的越权风险）：
- 由贵方后台用一个**双方共享的密钥**对 `{uid, token, exp}` 做 HMAC 签名，TripDesk 验签后才认 `uid`。
- 这样 `uid` 不可伪造、链接短时过期。具体协议我们对接时再定（一段约 30 行的签名逻辑）。

> 待我们确认贵方嵌入页的来源域名后，会把 TripDesk 的 `frame-ancestors` 锁定到该域名（目前为联调放开）。请提供将要承载 iframe 的页面域名。

---

## 6. 自测

仓库内 `test/embed-harness.html` 是一个模拟贵方后台的本地测试页：填 `base` / `uid` / `token` / `k`，点「载入 iframe」即可在本机验证嵌入与身份传递。

---

## 7. 我们需要贵方提供

1. 承载 iframe 的页面**域名**（用于锁定 `frame-ancestors`）。
2. `uid` 的取值口径（确认稳定、唯一、同一员工恒定）。
3. `token` 的注入方式确认（每次渲染 iframe 时取当前登录态的 travelkit token）。
4. 是否走签名握手（建议正式上线前启用）；如走，约定共享密钥的交换方式。
