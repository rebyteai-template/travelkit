# flight-verify ref

## flight_verify_solution

对用户选定的航班选项进行实时价格验证，确认价格有效后询问是否继续预订。

### 何时使用

用户从 `flight_search` 结果中选定 `1`/`2` 等数字标签后，立即调用此工具验价。

**不使用场景**：
- 搜索阶段（价格验证必须在用户选定选项后才触发）
- 用户尚未选择任何选项时

### 调用方式

内部使用选项标签对应的 `solutionId` 调用工具，**不向用户暴露 `solutionId`**。

### 验价一致性

调用 `flight_verify_solution` 后，核对返回结果是否对应用户选择的搜索选项：航班组合、路线、舱位/子舱位（工具返回时）和价格都应一致。

- 如果验价价格高于搜索展示价，或舱位/子舱位与展示选项不一致，不要继续预订；说明该价格方案已变化或映射不一致，并重新搜索/让用户重新选择最低价方案。
- 如果用户指出有更低价，或截图显示同路线/同航班组合有更低价，优先重新按同路线、日期、航班组合筛选最低价方案，再验价最低价对应选项。
- 不使用 `maxPrice` 作为默认搜索参数；只有用户给出价格上限、质疑价格，或需要核查更低价时才可辅助使用。

### 验证后展示内容

验价完成后，用普通语言展示：

- 最终价格（说明是否较搜索时有变化，工具结果支持时）
- 航班号和路线（航班只显示完整航班号，不展示航空公司名称；路线显示城市名 + 机场名 + IATA 代码 + 航站楼，**不能只显示 IATA 代码**）
- 出发和到达时间
- 舱位
- 行李额（数据返回时）
- 退改规则摘要（数据返回时）
- 低余票提醒（仅当工具明确返回任一航段 `availability` 小于等于 3 时，必须显示剩余票数）

退改规则若以 `*`、`>n`、`<n`、`-1`、`0` 等编码返回，先按 `output-rules` 的 Refund/Change Rule Codes 转成普通中文再展示；未返回规则、时间条件或金额时，不编造。

低余票提醒文案使用普通用户语言，并包含工具返回的剩余票数。单航段示例：`当前余票不多，仅剩 {availability} 张，请尽快完成预订和支付；未支付前票价和余票可能变化。` 多航段如果多个航段返回余票，使用最低余票数提醒，例如：`当前部分航段余票不多，最低仅剩 {minAvailability} 张，请尽快完成预订和支付；未支付前票价和余票可能变化。` 工具未返回余票时，不提示低余票；不要编造余票数量。

**常用机场展开对照**（仅 IATA 代码返回时扩展）：

| IATA | 中文名 |
|------|--------|
| PEK | 北京首都国际机场 |
| PKX | 北京大兴国际机场 |
| PVG | 上海浦东国际机场 |
| SHA | 上海虹桥国际机场 |
| BKK | 曼谷素万那普机场 |
| DMK | 曼谷廊曼机场 |

### 状态措辞规范

- ✅ 使用："实时价格验证通过"
- ❌ 不使用："座位已锁定"、"席位已占用"（除非工具明确返回锁定/占用状态）

**不暴露** `orderKey`。

### 验价失败处理

简短说明无法确认该选项，建议选择其他方案或重新搜索。

### 验价通过后

只询问用户是否继续预订，**不要**在这一阶段列出、改写、压缩或预告乘客信息字段：

> 实时价格验证通过，是否继续预订？确认后我会收集乘机人信息。

用户确认继续后，必须切换到乘客信息收集流程（参考 `flight-create-order` ref）。不要在 `flight_verify_solution` 流程内自行生成、补充或改写乘客信息提示词。不得提前添加生日示例、占位值，或把乘客类型选项收窄成当前搜索人数对应的类型。
